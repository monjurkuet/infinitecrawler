# PostgreSQL Migration Plan

## Goal

Replace MongoDB-based persistence with PostgreSQL-backed persistence for scraped data, using credentials from `.env`, while preserving the current scraper behavior and configuration-driven architecture.

## Current State

The codebase currently persists scraped data through two MongoDB output strategies:

- `strategies/output/mongodb.py`
  - `MongoDBOutputStrategy`: insert-only writes.
  - `MongoDBUpsertStrategy`: upserts by a configurable key field.
- `factory/scraper_factory.py`
  - Registers `mongodb` and `mongodb_upsert` strategy names.
- Config files and docs reference MongoDB directly:
  - `config/google_maps.yaml`
  - `config/gmaps_listings_working.yaml`
  - `docs/CONFIGURATION.md`
  - `docs/GMAPS_SEARCH_SCRAPER.md`
  - `docs/GMAPS_LISTINGS_SCRAPER.md`
  - `docs/ARCHITECTURE.md`
- `pyproject.toml`
  - No MongoDB dependency is currently declared, and there is no PostgreSQL dependency yet.
- `.env`
  - Contains `POSTGRESQL_HOST`, `POSTGRES_USERNAME`, and `POSTGRES_PASSWORD`.
  - The app does not currently load `.env` anywhere.

## Key Gaps To Address

1. There is no PostgreSQL output strategy yet.
2. Environment variables are not loaded into the process.
3. The PostgreSQL credential set appears incomplete for a full connection string:
   - missing obvious `database` name
   - missing obvious `port` (could default to `5432` if intended)
4. MongoDB-specific config keys (`uri`, `database`, `collection`) do not map cleanly to relational storage.
5. The current payloads are flexible dictionaries, so the schema design must handle variable fields safely.

## Recommended Migration Approach

Use a dedicated PostgreSQL output strategy that stores scraped items in a table with:

- relational metadata columns for operational fields
- a `JSONB` payload column for the scraped item body
- optional generated/updatable columns for the unique key used by upsert logic

This is the smallest correct change because it:

- preserves the current strategy/factory architecture
- avoids a large ORM/model refactor
- supports variable scraped payloads without requiring a rigid schema for every site
- maps naturally from the current dictionary-based item flow

## Proposed Storage Model

### Recommended PostgreSQL Names

Use the following defaults for the migration:

- database: `infinitecrawler`
- schema: `scraper`
- tables:
  - `scraper.gmaps_search_results`
  - `scraper.gmaps_listings`

Reasoning:

- `infinitecrawler` matches the project identity and avoids a generic database name like `postgres` or `scraping`
- `scraper` keeps application tables isolated from PostgreSQL system/default schema clutter
- separate tables preserve the current conceptual split between search-result harvesting and deep listing extraction

### Table Shape

Create one output table per configured logical destination. This is the least disruptive option and best matches the current config-driven design.

Recommended table columns:

- `id BIGSERIAL PRIMARY KEY`
- `key_value TEXT NULL`
  - stores the configured unique key value when present, e.g. `source_url`
- `source_type TEXT NULL`
  - optional classifier such as `gmaps_search` or `gmaps_listing`
- `payload JSONB NOT NULL`
  - stores the full scraped item dictionary
- `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
- `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`

Recommended indexes/constraints:

- unique index on `key_value` for upsert mode, filtered to non-null values if needed
- optional GIN index on `payload` if querying JSON fields is expected later

Recommended DDL pattern for upsert-capable tables:

```sql
CREATE SCHEMA IF NOT EXISTS scraper;

CREATE TABLE IF NOT EXISTS scraper.gmaps_search_results (
    id BIGSERIAL PRIMARY KEY,
    key_value TEXT,
    source_type TEXT,
    payload JSONB NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE UNIQUE INDEX IF NOT EXISTS gmaps_search_results_key_value_uidx
ON scraper.gmaps_search_results (key_value)
WHERE key_value IS NOT NULL;
```

### Why JSONB

The extractors emit flexible dictionaries and may differ across scraper configs. `JSONB` preserves this flexibility while still allowing PostgreSQL querying and indexing later.

## Implementation Steps

### 1. Add environment loading

Update startup so `.env` is loaded before strategies are initialized.

Likely change points:

- `main.py`

Plan:

- add `python-dotenv` dependency
- call `load_dotenv()` near process startup
- keep support for explicit config values where useful, but prefer env fallback for credentials

Expected outcome:

- PostgreSQL credentials in `.env` become available through `os.getenv(...)`

### 2. Add PostgreSQL dependency

Update `pyproject.toml` with a PostgreSQL client library.

Recommended dependency:

- `psycopg[binary]`

Reason:

- simple, reliable, modern PostgreSQL driver
- easy to use without introducing an ORM
- suitable for the current synchronous write pattern inside async methods

Alternative if stricter async DB usage is desired later:

- `asyncpg`

For this codebase, `psycopg` is the more minimal choice.

### 3. Implement PostgreSQL output strategies

Add a new module, likely `strategies/output/postgresql.py`.

Implement two strategies mirroring the existing MongoDB behavior:

- `PostgreSQLOutputStrategy`
  - insert-only behavior
- `PostgreSQLUpsertStrategy`
  - upsert behavior using configured `key_field`

Recommended config contract:

- `table`: destination table name
- `database`: PostgreSQL database name, defaulting to `infinitecrawler`
- `key_field`: unique item field used for upsert mode
- `max_results`: same behavior as current strategies
- optional overrides:
  - `host`
  - `port`
  - `user`
  - `password`
  - `database`
- `schema`

Connection resolution order:

1. explicit config value
2. `.env` value
3. hardcoded default only where safe, e.g. port `5432`

Suggested env mapping support:

- `POSTGRESQL_HOST` or `POSTGRES_HOST`
- `POSTGRES_USERNAME` or `POSTGRES_USER`
- `POSTGRES_PASSWORD`
- `POSTGRES_DB` or `POSTGRES_DATABASE`
- `POSTGRES_PORT`

If the database env var is absent, the implementation should default to `infinitecrawler` and log that default clearly during startup/connection setup.

Strategy responsibilities:

- connect on initialization
- create table if it does not exist
- create required index/constraint if configured for upsert mode
- serialize Python dicts to JSONB
- update `results_count`
- close DB connection during cleanup

### 4. Preserve current write semantics

Map the current MongoDB logic to PostgreSQL carefully.

For insert mode:

- current behavior: inserts every item
- PostgreSQL equivalent: `INSERT INTO ... (key_value, payload, created_at, updated_at) VALUES (...)`

For upsert mode:

- current behavior: upsert by `key_field` using full item replacement semantics
- PostgreSQL equivalent:
  - extract `item[key_field]` into `key_value`
  - `INSERT ... ON CONFLICT (key_value) DO UPDATE SET payload = EXCLUDED.payload, updated_at = NOW()`

Behavioral edge case to preserve:

- if `key_field` is missing, log an error and do not write

### 5. Register new strategies in the factory

Update `factory/scraper_factory.py` so the strategy map includes PostgreSQL entries.

Recommended names:

- `postgresql`
- `postgresql_upsert`

If you want seamless config migration with fewer user edits, you can optionally replace the old MongoDB strategy names in config at the same time. The cleaner long-term option is to update config names explicitly rather than aliasing MongoDB names to PostgreSQL behavior.

### 6. Migrate configuration files

Update configuration examples to reference PostgreSQL instead of MongoDB.

Files identified:

- `config/google_maps.yaml`
- `config/gmaps_listings_working.yaml`

Planned config transformation:

- `mongodb_upsert` -> `postgresql_upsert`
- `mongodb` -> `postgresql`
- replace `uri`, `database`, `collection` with PostgreSQL-oriented keys

Recommended example config shape:

```yaml
output:
  strategies:
    - strategy: "postgresql_upsert"
      config:
        database: "infinitecrawler"
        schema: "scraper"
        table: "gmaps_search_results"
        key_field: "source_url"
        source_type: "gmaps_search"
        max_results: 10000
    - strategy: "jsonl_file"
      config:
        file_path: "output/google_maps_{query}.jsonl"
        max_results: 10000
```

And for secondary output:

```yaml
secondary_output:
  strategy: "postgresql"
  config:
    database: "infinitecrawler"
    schema: "scraper"
    table: "gmaps_listings"
    source_type: "gmaps_listing"
    max_results: 100000
```

### 8. Decide whether MongoDB code is removed or retained temporarily

User intent says to remove MongoDB and use PostgreSQL instead. The implementation should therefore remove MongoDB from active usage, but there are two possible execution styles:

- hard replacement
  - delete `strategies/output/mongodb.py`
  - remove factory registrations
  - fully update docs/config
- soft transition
  - add PostgreSQL first, update all configs/docs, then delete MongoDB code in the same change

Recommended execution:

- soft transition during development within the same implementation branch, then finish with full removal before completion

This reduces breakage while editing, but the final state should not keep MongoDB as a supported output if the goal is full replacement.

### 9. Verification plan

After implementation, verify behavior end-to-end.

Recommended checks:

1. Dependency sync

4. Insert path

   - ensure existing file output still works unchanged
7. Cleanup path

If available, also run any project tests. No tests are currently obvious from the initial inspection, so lightweight runtime verification may be the main validation method.

## Risks And Mitigations

### 1. Missing PostgreSQL database name in `.env`

Risk:

- connection cannot be established without a database name unless a default is assumed

Mitigation:

- confirm the intended database name before implementation, or add a config/env fallback with a clearly documented default

### 2. Dynamic payloads may not fit a rigid relational schema

Risk:

- over-designing a table schema will slow implementation and make scrapers brittle

Mitigation:

- store full item data in `JSONB`

### 3. Async scraper methods using sync DB calls

Risk:

- synchronous PostgreSQL writes can block the event loop if throughput becomes very high

Mitigation:

- start with `psycopg` for minimal change
- if performance becomes an issue later, replace internals with `asyncpg` or move writes to a batching worker

### 4. Existing data migration from MongoDB may be needed

Risk:

- code migration alone does not preserve historical data

Mitigation:

- confirm whether old MongoDB data needs to be imported into PostgreSQL

## Resolved Decisions

1. Use PostgreSQL for future scraper runs only; no MongoDB historical-data migration is required.
2. Assume default PostgreSQL port `5432` unless config overrides it.
3. Use `infinitecrawler` as the default PostgreSQL database name.
4. Use `scraper` as the PostgreSQL schema.
5. Use `gmaps_search_results` and `gmaps_listings` as the initial table names.
6. Remove MongoDB code and documentation in the final implementation state.

## Recommended Final Implementation Scope

Unless you specify otherwise, the implementation should:

1. Load `.env` at startup.
2. Add `psycopg[binary]` and `python-dotenv` via `uv`-managed dependencies.
3. Add `postgresql` and `postgresql_upsert` output strategies using JSONB storage.
4. Update factory registration and scraper configs.
5. Replace MongoDB references in docs.
6. Remove MongoDB output code from the project.
7. Verify insert and upsert behavior against PostgreSQL.
