# Codebase Cleanup / Improve / Refactor Plan

## Objective

Raise reliability, maintainability, and operator usability of `infinitecrawler` with low-risk, incremental refactors.

Primary goals:
- Eliminate CLI/config pitfalls that cause failed or misleading runs.
- Improve runtime safety (queue recovery, cleanup, null-safe outputs).
- Reduce duplication across scrapers/strategies.
- Align docs with actual behavior and current schema.
- Add a minimal test harness for regression prevention.

---

## Current State Summary (Key Findings)

1. **Headless CLI behavior is broken/inconsistent**
   - `main.py` defines `--headless` as `store_true` with `default=True`, so it cannot be disabled (`main.py:67`).
   - Launcher passes `--no-headless` but `main.py` does not support it (`scripts/run_listing_crawlers.py:74-76`, `main.py:67-68`).

2. **Config shape inconsistency creates silent misconfig risk**
   - Strategies expect wrapped config (`{"config": {...}}`) e.g. JSONL output (`strategies/output/jsonl_file.py:16`).
   - Docs/examples include mixed output schema styles (`docs/CONFIGURATION.md`, `config/*.yaml`).

3. **Dynamic scraper can crash on missing output strategy**
   - It warns on missing output, but later calls methods without always guarding every usage path (`scrapers/dynamic_scraper.py:303`, `:363`).

4. **Queue stall recovery exists but is unused**
   - `requeue_stalled()` implemented but not called from processing loops (`strategies/queue/redis_queue.py:151-179`).

5. **Input DB strategy has no cleanup lifecycle**
   - PostgreSQL input opens connection in constructor and never closes (`strategies/input/postgresql_uncrawled.py:31`).

6. **Duplication and drift risks**
   - Similar browser lifecycle and strategy re-init logic split across both scrapers.
   - Duplicate launcher scripts with overlapping concerns.
   - Repeated PostgreSQL settings resolution logic in input/output strategies.

7. **Documentation mismatches**
   - CLI examples still include invalid patterns (e.g. boolean arg style), and some architecture wording does not match actual runtime behavior.

---

## Refactor Principles

- **Small, reversible PR-sized steps** (no big-bang rewrite).
- **Behavior-preserving first**, then structural improvement.
- **Add guardrails before abstractions** (validation + tests first).
- **Prefer schema normalization layer** over touching every existing config immediately.
- **Do not change scraping semantics unless explicitly intended**.

## Execution Plan (Phased)

## Phase 1 - Stabilize Operator Experience (High Value, Low Risk)

### Scope
- Fix CLI headless flags and launcher compatibility.
- Update docs/README command examples.
- Add command-line smoke tests.

### Tasks
1. Update `main.py` argparse:
   - Replace `store_true` with `argparse.BooleanOptionalAction` for `--headless/--no-headless`.
2. Ensure launcher parity:
   - Keep `scripts/run_listing_crawlers.py` behavior aligned with `main.py`.
3. Doc fixes:
   - Normalize examples to `uv run python main.py ... --no-headless` where needed.
4. Add CLI smoke test file:
   - Validate parser accepts both headless flags and expected defaults.

### Acceptance Criteria
- `python main.py --help` shows both `--headless` and `--no-headless`.
- Launcher runs with headed mode without arg errors.
- Docs no longer include invalid boolean arg examples.

### Effort
S

---

## Phase 2 - Configuration Safety and Strategy Guardrails

### Scope
- Add config normalization/validation at load time.
- Make output handling null-safe and deterministic.

### Tasks
1. Add a config normalization layer in factory (or dedicated module):
   - Support both `output.strategy/config` and legacy/flat forms.
   - Normalize browser fields (`browser_automation` vs nested `browser.automation`) into one internal shape.
2. Add explicit validation with actionable errors:
   - Missing required fields per `content_type`.
   - Unknown strategy names.
3. Introduce `NullOutputStrategy`:
   - No-op writes, `has_reached_limit=False`, safe cleanup.
   - Inject when output is omitted to avoid `None` path crashes.
4. Harden `DynamicScraper` and `ListingCrawler` for output calls:
   - Use interface methods only; avoid implicit assumptions.

### Acceptance Criteria
- Misconfigured output yields clear startup error or safe no-op behavior (as designed).
- Existing configs continue to run through normalization.
- No `AttributeError` when output is missing.

### Effort
M

---

## Phase 3 - Runtime Reliability (Queue + Resource Lifecycle)

### Scope
- Prevent stranded work in Redis processing queue.
- Ensure DB input connections are cleaned up.

### Tasks
1. Integrate stalled requeue checks in queue loops:
   - Call `requeue_stalled()` periodically in both scrapers (time-based or every N iterations).
2. Extend strategy lifecycle contract:
   - Add optional `cleanup()` to `InputStrategy` interface (default no-op in base pattern).
3. Implement `cleanup()` for `PostgreSQLUncrawledInputStrategy`.
4. Call input/queue/output cleanup consistently in scraper `cleanup()`.

### Acceptance Criteria
- Simulated worker interruption does not leave URLs permanently stuck in `processing`.
- DB connections close on graceful shutdown/restart.

### Effort
M

---

## Phase 4 - Structural Refactor to Reduce Duplication

### Scope
- Consolidate duplicated logic and create shared helpers.

### Tasks
1. Extract shared scraper lifecycle helper/mixin:
   - Browser start/restart/cleanup pattern.
   - Browser-bound strategy refresh pattern.
2. Consolidate launcher scripts into one canonical launcher.
3. Extract PostgreSQL settings resolver utility:
   - Shared env + config precedence logic.
4. Reduce abstraction leakage:
   - Avoid direct use of strategy-specific properties not in interface (e.g., `max_results` access from scraper logic).

### Acceptance Criteria
- Less duplicate code paths across scrapers.
- One launcher script serves all current launch patterns.
- PG connection setup behavior is consistent across input/output.

### Effort
M

## Phase 5 - Documentation and Developer UX Hardening

### Scope
- Make docs an accurate reflection of implementation.
- Add maintainability checks.

### Tasks
1. Update `docs/CONFIGURATION.md`, `docs/ARCHITECTURE.md`, and README:
   - Real runtime model for worker/process behavior.
   - Canonical schema examples (normalized format).
2. Add a “config migration notes” section:
   - Legacy keys accepted and deprecation timeline.
3. Add lint/type/test quickstart section using `uv` only.
4. Optional: add a small script to validate all `config/*.yaml` files against schema.

### Acceptance Criteria
- Docs and code behavior are aligned.
- New users can run with only README steps and no trial/error for flags/schema.

### Effort
S-M

## Test Strategy

Minimum test set to add during refactor:

1. **CLI parser tests**
   - `--headless`, `--no-headless`, default behavior.

2. **Config normalization tests**
   - Legacy output shape -> normalized shape.
   - Missing required keys -> clear errors.

3. **Scraper safety tests**
   - No output configured -> no crash.
   - Unknown strategy -> deterministic error.

4. **Queue reliability tests**
   - Stalled processing entries are requeued.

5. **Lifecycle cleanup tests**
   - Input/output cleanup invoked on scraper shutdown.

## Risk Register and Mitigations

- **Risk:** Breaking existing YAML configs during normalization.
  - **Mitigation:** Compatibility adapter + warning logs + migration notes.

- **Risk:** Queue requeue logic causing duplicate processing.
  - **Mitigation:** conservative timeout defaults + idempotent output strategies (already upsert-capable for PG).

- **Risk:** Refactor churn in scraper lifecycle.
  - **Mitigation:** preserve public interfaces, move code gradually, add tests before extraction.

- **Risk:** Doc drift reappears.
  - **Mitigation:** add doc check item to PR template and keep one canonical schema source.

## Definition of Done

- No known CLI flag mismatch between entrypoint and scripts.
- Core configs in `config/` run without schema ambiguity.
- Scrapers are resilient to missing optional components.
- Queue can recover from interrupted workers.
- Long-running sessions do not leak input DB connections.
- Docs reflect actual behavior and are executable as written.
