# Google Maps Listing Details Persistence Plan

## Goal

Store Google Maps listing-detail scraper output in PostgreSQL using a purpose-built schema for `gmaps_listings`, while preserving raw payload storage for resilience and future extraction changes. At the same time, clean up the current listing-crawler persistence flow so reruns, deduplication, and operational behavior are more predictable.

Additional goal:

- allow the listing crawler to fetch only uncrawled Google Maps listing URLs directly from PostgreSQL search results, without requiring a manually exported `output/gmaps_urls.txt` file
- enable running 4 listing crawler instances in parallel to increase extraction throughput

## Current State

- Search results already persist to `scraper.gmaps_search_results` through `PostgreSQLUpsertStrategy`.
- Listing details currently write through `secondary_output.strategy: postgresql` into `scraper.gmaps_listings` using the generic insert-only strategy.
- The current PostgreSQL output shape is generic:
  - `id BIGSERIAL`
  - `key_value TEXT`
  - `source_type TEXT`
  - `payload JSONB`
  - `created_at TIMESTAMPTZ`
  - `updated_at TIMESTAMPTZ`
- Listing detail writes are not currently upserts, and the config does not provide a `key_field` for detail deduplication.
- `ListingCrawler` adds `_crawl_meta.source_url` to the payload, and `multi_step` extracts `place_id`, `latitude`, `longitude`, and content fields, which gives enough information to design a real details table.

## Problems To Solve

1. `scraper.gmaps_listings` is currently too generic for operational querying.
   - Important fields like `place_id`, `name`, `rating`, `review_count`, `phone`, `website`, and `address` are buried only in JSONB.
2. Listing detail persistence is insert-only.
   - Re-running the same URL can create duplicate rows instead of refreshing the current business snapshot.
3. Search results and listing details are not explicitly linked at the relational layer.
   - The shared `source_url` and extracted `place_id` are present, but there is no designed join contract.
4. Type fidelity is weak.
   - Numeric-like values such as `rating`, `review_count`, `latitude`, `longitude`, and boolean `is_claimed` should be queryable as real SQL types.
5. Listing-crawler restart behavior is incomplete.
   - `_restart_browser()` only restarts the browser; it does not reinitialize extraction/navigation state against the new browser instance.
6. Documentation drift still exists in the listing-detail path.
   - `docs/GMAPS_LISTINGS_SCRAPER.md` still references `pymongo` in Python dependencies.
7. Listing URL sourcing is still manual.
   - The listing crawler currently depends on `file_url_loader` and `output/gmaps_urls.txt`, even though the required URLs already exist in `scraper.gmaps_search_results`.
8. "Uncrawled" is not modeled as a first-class input source yet.
   - The natural definition is "search-result URLs that do not yet have a matching row in `scraper.gmaps_listings`", but current code has no strategy for that query.
9. Redis queue state can conflict with database truth.
   - Even if a URL is uncrawled in PostgreSQL, `RedisQueueStrategy.enqueue()` will still skip it if the URL remains in Redis `completed` from a previous run.
10. Listing-detail parsing is currently too generic and retry-heavy.
   - Broad selectors like `button` for category and repeated retries for optional fields waste significant time per listing.
11. The current extractor does not take advantage of stable Google Maps DOM attributes.
   - The provided DOM sample shows stable targets like `h1.DUwDvf`, `div.F7nice`, `button[data-item-id="address"]`, `a[data-item-id="authority"]`, `button[data-item-id^="phone"]`, and `button[data-item-id="oloc"]`.
12. Optional fields are treated too much like required fields.
   - Missing fields such as `booking_url`, `website`, and sometimes `review_count` currently burn the full retry/backoff path and make the crawler appear stuck.
13. `workers.count` does not currently create concurrent listing workers inside one process.
   - `ListingCrawler.scrape()` starts one browser, loads URLs once, and calls a single `_process_queue()` loop.
14. Throughput is currently limited to one active browser session per process.
   - Even with parser improvements, each listing still incurs page load, tab interaction, and rate limiting, so the fastest safe throughput gain is multiple crawler processes competing on the same Redis queue.
15. Multi-instance operability is not yet documented or first-class.
   - The current code can support shared-queue consumption, but there is no implementation plan yet for instance naming, logging clarity, or a one-command launcher for 4 parallel crawlers.

## Recommended Design

### Database Objects

Keep the schema as `scraper` and use two main application tables:

1. `scraper.gmaps_search_results`
2. `scraper.gmaps_listings`

No historical migration is required. The existing `scraper.gmaps_listings` table can be recreated cleanly during implementation.

### Recommended `scraper.gmaps_listings` Table Shape

Use a hybrid schema: normalized top-level columns for common query fields plus a `payload JSONB` column containing the full raw extracted record.

Recommended columns:

- `id BIGSERIAL PRIMARY KEY`
- `place_id TEXT`
- `source_url TEXT NOT NULL`
- `key_value TEXT NOT NULL`
- `source_type TEXT`
- `name TEXT`
- `category TEXT`
- `rating NUMERIC(3,2)`
- `review_count INTEGER`
- `address TEXT`
- `phone TEXT`
- `website TEXT`
- `booking_url TEXT`
- `plus_code TEXT`
- `is_claimed BOOLEAN`
- `latitude DOUBLE PRECISION`
- `longitude DOUBLE PRECISION`
- `crawl_retry_count INTEGER`
- `crawl_pages_processed INTEGER`
- `payload JSONB NOT NULL`
- `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
- `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`

### Uniqueness / Upsert Key

Use a deterministic unique key for listing details:

- Preferred key: `place_id`
- Fallback key: `source_url`

Recommended implementation approach:

- Persist `key_value` as:
  - `place_id` when present
  - otherwise `source_url`
- Add a unique index on `key_value`

This keeps the strategy generic enough while still giving reliable upsert behavior.

### Index Recommendations

Create these indexes for `scraper.gmaps_listings`:

- unique index on `key_value`
- btree index on `place_id`
- btree index on `source_url`
- btree index on `name`
- btree index on `category`
- btree index on `updated_at DESC`
- optional GIN index on `payload` if JSONB ad hoc querying is expected

### Relationship To Search Results

Do not introduce a hard foreign key in the first implementation. Instead, standardize the join path through shared identifiers.

Recommended join contract:

- `gmaps_search_results.key_value = source_url` today for search data
- `gmaps_listings.source_url` stores the same Google Maps listing URL
- `gmaps_listings.place_id` gives a more stable business identifier for future uses

Reasoning:

- Search results currently upsert by `source_url`
- Listing details originate from those URLs
- A hard FK would complicate writes if search and detail rows are loaded independently

### Uncrawled Listing Input Source

Add a PostgreSQL-backed input strategy for listing crawling.

Recommended strategy name:

- `postgresql_uncrawled_gmaps`

Responsibility:

- query `scraper.gmaps_search_results` for listing URLs
- left join against `scraper.gmaps_listings`
- return only URLs that do not yet have a corresponding listing-details row
- deduplicate URLs before yielding them

Recommended uncrawled definition:

- `gmaps_search_results.payload->>'source_url' IS NOT NULL`
- and there is no matching `gmaps_listings.source_url`
- treat any existing `gmaps_listings.source_url` row as already crawled, regardless of payload completeness

Representative query shape:

```sql
SELECT DISTINCT sr.payload->>'source_url' AS source_url
FROM scraper.gmaps_search_results sr
LEFT JOIN scraper.gmaps_listings gl
  ON gl.source_url = sr.payload->>'source_url'
WHERE sr.payload->>'source_url' IS NOT NULL
  AND gl.source_url IS NULL;
```

Optional future refinement:

- allow filtering by `query`, `created_at`, or search-result source metadata
- do not implement those filters unless needed during implementation

### Listing Parser Optimization

Use the observed Google Maps detail DOM to move from broad, retry-heavy extraction to targeted, low-latency parsing.

Key principle:

- prefer stable semantic attributes and compact containers over generic tag selectors
- treat missing optional fields as normal, not as retry-worthy failures

Recommended field extraction targets from the sample DOM:

- `name`
  - primary: `h1.DUwDvf`
- `subtitle` or local-language title (optional future field)
  - `h2.bwoZTb`
- `rating`
  - primary container: `div.F7nice`
  - preferred extraction: descendant `[aria-hidden="true"]` or `span.ceNzKf[aria-label*="stars"]`
- `review_count`
  - primary: `div.F7nice span[role="img"][aria-label*="reviews"]`
  - parse commas from values like `1,158 reviews`
- `category`
  - primary: `button[jsaction*="category"]`
  - avoid broad `button` selector
- `address`
  - primary: `button[data-item-id="address"] .Io6YTe`
- `website`
  - primary: `a[data-item-id="authority"]`
  - attribute: `href`
- `phone`
  - primary: `button[data-item-id^="phone"] .Io6YTe`
- `plus_code`
  - primary: `button[data-item-id="oloc"] .Io6YTe`
- `is_claimed`
  - keep as optional presence check, but do not retry heavily
- `hours_summary` or `current_hours` (optional future field)
  - `div.OqCZI .ZDu9vd`

DOM-informed review/about navigation observations:

- the sample tab order is `Overview`, `Reviews`, `About`
- current config assumes `Overview`, `About`, `Reviews`
- tab selection should prefer label/aria matching over `nth-of-type` where possible

Recommended navigation selector strategy:

- Overview: `[role="tab"][aria-label^="Overview"]`
- Reviews: `[role="tab"][aria-label^="Reviews"]`
- About: `[role="tab"][aria-label^="About"]`

This is more robust than positional tab indexing.

### Parallel Extraction Model

Use process-level parallelism, not in-process asyncio worker fanout, as the primary scaling model for listing extraction.

Why this is the recommended design:

- `RedisQueueStrategy.dequeue()` uses atomic `BRPOPLPUSH`, so multiple independent processes can safely compete for URLs from the same queue
- each `ListingCrawler` currently owns exactly one browser manager and one extraction pipeline
- browser automation is the dominant bottleneck, so multiple browser processes give real parallelism without a large refactor
- the existing codebase does not yet implement `workers.count` as internal concurrent workers, so process-level parallelism is the smallest correct change

Recommended target behavior:

- run 4 independent `uv run python main.py --config config/gmaps_listings_working.yaml` processes in parallel
- all 4 attach to the same Redis queue keys
- each process dequeues different URLs atomically
- all 4 write to the same PostgreSQL `scraper.gmaps_listings` table via upsert

Expected impact:

- best-case throughput approaches roughly 4x minus browser/resource contention
- practical throughput will depend on CPU, RAM, Chrome stability, network latency, and Google Maps anti-automation behavior

### Resource and Contention Considerations

Before implementing 4-process support, account for these constraints:

- each crawler instance launches its own Chrome browser process and user-data dir
- 4 instances will multiply:
  - CPU usage
  - memory usage
  - outbound request rate
  - chance of hitting Google Maps throttling or transient UI failures
- PostgreSQL writes are safe because listing writes are upserts keyed by `place_id`/`source_url`
- Redis queue safety is already acceptable for multi-process consumption

Recommended operational defaults for 4-instance mode:

- keep the same queue keys shared across all instances
- keep PostgreSQL upsert enabled
- maintain modest request delays to avoid overwhelming Maps
- add instance labeling in logs so one can distinguish crawler 1/2/3/4

### Parallel Launch Strategy

Two implementation options exist.

Option A: external process launcher script or command wrapper

- add a helper command/script that launches 4 identical crawler processes
- example shape:
  - one parent process spawns 4 children
  - each child runs the same config against the shared Redis queue

Pros:

- smallest code change to crawler internals
- aligns with current queue design
- easiest to reason about failures and restarts

Cons:

- requires a launcher entrypoint or shell command for convenience
- per-instance logs should be labeled explicitly

Option B: in-process worker fanout inside `ListingCrawler`

- implement `workers.count` by spinning up N browser managers and N concurrent worker tasks inside one process

Pros:

- one command, one parent process
- direct use of the existing `workers.count` config field

Cons:

- much larger refactor
- browser manager, extraction strategy, and delay manager state would all need per-worker isolation
- more risk of subtle concurrency bugs

Recommended choice:

- implement Option A first
- keep `workers.count` for future true in-process concurrency if ever needed
- implement the first version as a launcher script/command that starts 4 labeled crawler processes automatically

## Recommended Code Changes

### 1. Create a dedicated PostgreSQL upsert strategy for listing details

Add a specialized output strategy, preferably in `strategies/output/postgresql.py`, for example:

- `PostgreSQLListingDetailsUpsertStrategy`

Behavior:

- Read the same constructor config shape as existing output strategies
- Derive a deterministic `key_value` from `place_id` or `_crawl_meta.source_url`
- Extract common columns from the item payload into typed SQL columns
- Preserve the full raw item in `payload JSONB`
- Use `INSERT ... ON CONFLICT (key_value) DO UPDATE`
- Refresh mutable fields and `updated_at` on rerun

Why a specialized strategy is recommended instead of only extending the generic one:

- listing-detail rows need column mapping and type conversion
- it avoids overcomplicating the shared generic insert strategy used elsewhere
- it keeps search-result storage and detail storage independently evolvable

### 2. Register the new strategy in `factory/scraper_factory.py`

Add a factory mapping such as:

- `postgresql_listing_upsert`
- `postgresql_uncrawled_gmaps`

Keep the current lazy-import pattern intact.

### 3. Update listing config to use the new strategy

Change `config/gmaps_listings_working.yaml`:

- replace `input.strategy: file_url_loader`
- with `input.strategy: postgresql_uncrawled_gmaps`
- replace manual file-path based listing input with database-backed uncrawled selection
- keep `secondary_output.strategy: postgresql_listing_upsert`

Recommended input config block:

```yaml
input:
  strategy: "postgresql_uncrawled_gmaps"
  config:
    database: "infinitecrawler"
    schema: "scraper"
    search_results_table: "gmaps_search_results"
    listings_table: "gmaps_listings"
    source_url_field: "source_url"
    batch_size: 1000
```

Recommended config block:

```yaml
secondary_output:
  strategy: "postgresql_listing_upsert"
  config:
    database: "infinitecrawler"
    schema: "scraper"
    table: "gmaps_listings"
    key_field: "place_id"
    source_type: "gmaps_listing"
    max_results: 100000
```

Implementation note:

- The strategy should still fall back to `_crawl_meta.source_url` if `place_id` is missing, regardless of `key_field`.

### 3a. Align queue semantics with database-backed uncrawled selection

Current issue:

- `RedisQueueStrategy.enqueue()` always skips URLs already present in Redis `completed`
- that makes Redis state override PostgreSQL truth

Recommended implementation:

- keep Redis queue mechanics as-is for pending/processing/failure handling
- add an opt-in queue config flag for listing crawling, for example:
  - `ignore_completed_on_enqueue: true`
- when enabled, `enqueue()` should skip only URLs already in `pending`, and let the PostgreSQL input strategy be the source of truth for whether a URL is uncrawled

Reasoning:

- the database already knows whether details exist
- otherwise the user must keep clearing Redis manually even though PostgreSQL is authoritative

Alternative implementation if queue changes are to be minimized:

- have the listing crawler clear its own queue completion set before enqueueing database-derived uncrawled URLs

Preferred choice:

- implement the queue flag instead of implicit destructive Redis cleanup

### 4. Improve `ListingCrawler` browser restart handling

Current issue:

- `_restart_browser()` restarts only the browser manager.
- `MultiStepExtractionStrategy` and navigation strategy instances were created with the old browser manager reference.

Plan:

- after restarting the browser, re-run `initialize_strategies()` or at minimum reinitialize browser-bound strategies
- ensure output strategies are not duplicated/leaked during strategy reinit

This is a meaningful cleanup item discovered while reviewing the listing-detail flow.

### 5. Recreate the listing-details table

Implementation should recreate `scraper.gmaps_listings` for the new schema.

Because user intent is forward-looking persistence rather than historical migration, implementation should:

- drop and recreate `scraper.gmaps_listings` if it already exists
- create the final typed-column + JSONB schema from scratch
- recreate the required indexes after table creation

This keeps the implementation simpler and avoids carrying forward the generic insert-only table shape.

## Detailed Schema Recommendation

Recommended DDL target:

```sql
CREATE SCHEMA IF NOT EXISTS scraper;

CREATE TABLE IF NOT EXISTS scraper.gmaps_listings (
    id BIGSERIAL PRIMARY KEY,
    place_id TEXT,
    source_url TEXT NOT NULL,
    key_value TEXT NOT NULL,
    source_type TEXT,
    name TEXT,
    category TEXT,
    rating NUMERIC(3,2),
    review_count INTEGER,
    address TEXT,
    phone TEXT,
    website TEXT,
    booking_url TEXT,
    plus_code TEXT,
    is_claimed BOOLEAN,
    latitude DOUBLE PRECISION,
    longitude DOUBLE PRECISION,
    crawl_retry_count INTEGER,
    crawl_pages_processed INTEGER,
    payload JSONB NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE UNIQUE INDEX IF NOT EXISTS gmaps_listings_key_value_uidx
    ON scraper.gmaps_listings (key_value);

CREATE INDEX IF NOT EXISTS gmaps_listings_place_id_idx
    ON scraper.gmaps_listings (place_id);

CREATE INDEX IF NOT EXISTS gmaps_listings_source_url_idx
    ON scraper.gmaps_listings (source_url);

CREATE INDEX IF NOT EXISTS gmaps_listings_name_idx
    ON scraper.gmaps_listings (name);

CREATE INDEX IF NOT EXISTS gmaps_listings_category_idx
    ON scraper.gmaps_listings (category);

CREATE INDEX IF NOT EXISTS gmaps_listings_updated_at_idx
    ON scraper.gmaps_listings (updated_at DESC);
```

Optional later index:

```sql
CREATE INDEX IF NOT EXISTS gmaps_listings_payload_gin_idx
    ON scraper.gmaps_listings USING GIN (payload);
```

## Data Mapping Plan

Map extracted fields into columns as follows:

- `place_id <- item.get("place_id")`
- `source_url <- item.get("_crawl_meta", {}).get("source_url") or item.get("source_url")`
- `key_value <- place_id or source_url`
- `source_type <- config.source_type or item.get("source")`
- `name <- item.get("name")`
- `category <- item.get("category")`
- `rating <- parsed numeric from item.get("rating")`
- `review_count <- parsed int from item.get("review_count")`
- `address <- item.get("address")`
- `phone <- item.get("phone")`
- `website <- item.get("website")`
- `booking_url <- item.get("booking_url")`
- `plus_code <- item.get("plus_code")`
- `is_claimed <- item.get("is_claimed")`
- `latitude <- parsed float from item.get("latitude")`
- `longitude <- parsed float from item.get("longitude")`
- `crawl_retry_count <- item.get("_crawl_meta", {}).get("retry_count")`
- `crawl_pages_processed <- item.get("_crawl_meta", {}).get("pages_processed")`
- `payload <- full serialized item`

Type conversion rules:

- empty strings become `NULL`
- malformed numeric values become `NULL` rather than failing the whole row
- booleans are preserved directly

## Decision Summary

- Canonical database remains PostgreSQL.
- Schema remains `scraper`.
- Canonical table for listing details remains `scraper.gmaps_listings`.
- Persistence model should be hybrid: typed columns + `payload JSONB`.
- Listing details should use upsert semantics, not insert-only semantics.
- Upsert key should be `place_id` when available, otherwise `source_url`.
- Listing crawler input should come directly from PostgreSQL search results, not a manually exported text file.
- "Uncrawled" should mean: search-result `source_url` exists and no matching `gmaps_listings.source_url` exists yet.
- Redis queue state should not override PostgreSQL uncrawled selection for listing crawling.
- Listing-detail parsing should prefer stable semantic selectors (`data-item-id`, `aria-label`, known containers) over broad generic selectors.
- Optional fields should not consume the full retry/backoff path by default.
- Parallelism should be implemented first as 4 independent crawler processes sharing one Redis queue, not as in-process worker fanout.
- `workers.count` should not be treated as already-functional concurrency for listing crawling.
- No historical migration or snapshot table is required for this phase.
- Include listing-crawler restart fix and doc cleanup in the same implementation pass.
